Get Involved

Spiffy UI is an open source project with a supportive community. We welcome coders, testers, documenters, and anyone else with good ideas for the framework.

Give us feedback

The easiest way to get involved is to give us some feedback. Have a great idea for a new feature? A cool way of getting something done? Let us know about it on the Spiffy UI Google Group.

Report bugs

If you find bugs we want to hear about them and fix them. Please file bug reports and let us know if you find what you find.

Not sure if something is a bug? You can ask about it on the Spiffy UI Google Group.

Share the framework

Another great way to help out is to talk about us on Twitter, Facebook, and any other social sharing service. The more people who know about Spiffy UI the stronger our community will become.

For special bonus points use the #spiffyui hashtag when you talk about us on Twitter.

Write an article

If you like Spiffy UI and use it in your project we would love to hear about it. Write an article about Spiffy UI and we'll link to it from our project home page.

Contribute

Can you sling code like a pro? Why not help us out with Spiffy UI. We're always happy to take patches, bug fixes, and new features.

Once you've checked out the code and made sure your change builds you can generate a patch. We always love patches, but we're kind of picky about our code. For a patch to get accepted it must:

Build and run. This sounds obvious, but we need all code to build and run on all of our supported browsers listed on the before we can take it.

Well formatted. All Java code must pass 100% percent of all Checkstyle checks and all JavaScript code must pass JSLint with no warnings. Both of these are included in the build.

Be generic. Projects like this get bloated really fast if we're not careful. Everything we take into the project must be generically usable.

Be bounded. Spiffy UI is dedicated to modularity. Every piece of code must have clear boundaries.

Be internationalized. All strings must be in resource bundles, all dates and currencies must be localizable, and all encodings should be UTF-8.

Be independent. Spiffy UI runs against REST endpoints written on any server so your contribution can't rely on any Java on the server.

Well documented. Every class and interface in Spiffy UI must have well written and complete JavaDoc.

Fast. Spiffy UI wants to make every page load in under 1.5 seconds. That means each component must be fast.

Unit tested. All new code should include unit tests where appropriate.

Still interested in contributing? Once you have a patch you can email it to us or post it to our Review Board server.

ReviewBoard

ReviewBoard is a convenient way to contribute patches, using a code review web application that improves the conversation about the contribution.

Register an account for yourself on the ReviewBoard server.

Submit a new review request using either the Web application and your prepared patch, or from the command-line using the post-review tool.

In either situation, make sure that the SpiffyUI repository is selected and that the spiffyui Review Group is added to the new review request.
